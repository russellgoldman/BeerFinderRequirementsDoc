1. Introduction


1.1 The Purpose of the Document

The purpose of this document will be to outline and describe the application, the application interface requirements, design requirements, business and marketing plan and finally the implementation process.


1.2 Scope of the Document

Our application will be web based. It will require that a user will have the ability to use a computer, connect to the internet (Wi-Fi or wired ethernet connections), and use a web browser to navigate through our application site.


1.3 Overview of the Application

This section describes the software requirements for the BeerBuddy application. The purpose of our application will be to make (preferred) beer acquisition easier and faster. The application aims to increase the speed of the buying process. Presently, customers go through a six-step process when purchasing anything. In order; problem recognition, information search, evaluation of alternatives, purchase decision, purchase and post purchase evaluation. Essentially the problems a consumer faces are; they don’t know what they want, they must work and evaluate choices (costly endeavor) and finally they are left to review themselves whether they like the products/services they purchased. Our web application will allow users to use essential primary criteria to make choices that will best suit their preferences. Since the consumer already has a historic previous preference they will be able to make the choices that will offer them relevant options to choose from instead of testing out every product to determine if they enjoy it or not.



2. Business Context


2.1 Ease of Use

This section describes the consumer view for the BeerBuddy application. Initially the user will have the main screen with two options. An option where they choose whether they are familiar with beers, the experience option, or they are novice beer consumers, the novice option. Choosing the experienced option will then lead to a new page that will be selection criteria. After the user chooses their preferences, the application will generate a list of relevant beers. The user will then be able to select the beer they would like based on the entered preferences. The latter option of novice consumers will lead to a page which will have approximately five to ten beers which are extremely different. The user will be able to select the one that interests them, the application will list the qualities or key characteristics of the drink and any relevant reviews. For all the drinks the assumption will be made that they are available for purchase locally (for this project local is assumed to be Waterloo Region).


2.2 Scalability

This section describes the scalability for the BeerBuddy application. Initially, the application is designed to be accessible via websites. As a business model, this application would ideally made for tablets which would be available at local bars, pubs, and nightclubs. This would make it easier for people to pick out new drinks or similar drinks to try. If implemented in tablet mode, the application could also be used in local beer stores or stores with an assortment of beers/alcohols that are available for purchase. This would increase the stores revenue and the data can then be recorded and sold to beer/alcohol manufacturers so that they can monitor what the consumers like to drink. More generally the algorithm could also be applied to other products to determine brand/quality preferences.



3. Current System Description


3.1 Purpose

The goal of BeerBuddy is to provide insight and advice in beer choices of its users. They can enter criteria that they want to use to query related beers, as well as adding custom beers to the database (would involve use of API). The service assumes that the beers are available locally (same city as one resides in).


3.2 User Characteristics

User must understand terms about beer



Select Criteria

Option to choose whether they are a novice consumer of beer or not

Pick Criteria Attributes

Pick preferences of what kind of beer they like or enjoy. What combination of attributes do they prefer.

View Queried Beers

A list of relevant beers based on the attributes selected

Add Custom Beer

Incase preferred beer is not found, or user has already tried all the listed suggested beers. A window is available (through an “Add New Beer” button) for the user to enter a new beer that fits the selection criteria previously listed (if not selected the user will enter attributes on their own). The user needs to have an active account in order to be able to add the beer to the database.



4. Functional Requirements


4.1 Beer Search Optimization

  • User enters information about beer through tags/search filters
  • Produce search results
  • Search results will be sorted based on options entered by user or default option

4.2 Reviews

  • User posts opinions of beers
  • User rates beers
  • Forum page for each product for discussion

5. Interface Requirements


5.1 Selection Screens (web-based)

Users select the criteria they want, adjust the attributes of that property, and then choose to change criteria or query the database. These screens will have intuitive selective features such like drop down menus and text field inputs.



6. Performance Requirements


6.1 Hardware Requirements

This application requires a computer or smartphone that connects with internet and must have an internet browser.


6.2 Software Requirements

This application will require that the device have an ability to connect to the internet.


6.3 Human Capability Requirement

This application will require that the user be able to understand the instructions provided and have a notion of beer tastes and qualities unless they are a novice and choose the novice option.



7. Design Requirements


7.1 Standards Compliance

The project members are expected to follow the Laurier standard of application designs. This includes the documentation and program design conventions and practices. If presented with a better alternative or a Laurier Computer Science approved standard (i.e. JavaDoc) the team will render the project according to that standard providing that sufficient benefit is observed and agreed upon by the group.

10. Portability


  • The service needs to be accessible on all browsers (Chrome, Firefox, Microsoft Edge, Opera, Safari, Internet Explorer (if time permits), etc). Since the application will be Bootstrapped, mobile responsiveness will be built-in with the app. The service will be responsive to all current versions of web browsers.


11. Reusability


  • The application is assumed to be open source, available for public use. Any programmer will be allowed to reuse part or all the code during the preliminary development stage. This group reserves the right to refuse the open source availability of the application code.


12. Optional Features


  • Databases can be updated with information from (local) beer vendors with price, sales, etc.
  • If an individual who lacks knowledge of beer wants to use the service, we will implement a help feature button which explains the types of beers to help the user get a general understanding of their differences.
  • A review feature, time permitting, will be attempted so that beer consumers can rank their personal favorites.
  • Future scaling option to create login mechanism for reviewing purposes. Users of the application will be able to have an account that will store all their pending or approved purchases.
